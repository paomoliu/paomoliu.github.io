git基础

  • git

在大一下的一门选修课上第一次接触到git，当时所有的操作都是在Linux系统下进行，觉得十分的深奥。在后面的学习中，偶尔会用一用一些基本的命令，最近发现关于git的一些内容都忘了一些，于是重新拾起当时老师讲的课件，做一个复习总结，也方便自己以后查阅。

本章介绍使用git前的基础知识，了解版本控制系统（VCS），git项目的三个区域。下一篇总结git的常见用法。

1. 什么是版本控制？

版本控制是一种用于跟踪文件版本的系统，可以记录若干文件的内容变化，以便将来查阅特定版本修订情况。我们可以对任何类型的文件进行版本控制。

如果你是位图形或网页设计师，可能会需要保存某一幅图片或页面布局文件的所有修订版本。采用版本控制系统是个明智的选择。有了它就可以将某个文件回溯到之前的状态，甚至将整个项目回退到过去某个时间点的状态。

许多人习惯用复制整个项目目录的方式来保存不同的版本，或许还会改名加上备份时间以示区别。这么做唯一的好处就是简单，不过坏处却不少：有时会混淆所在的工作目录，弄错了文件丢了数据就没了后退的路。

1.1 早期的版本控制

在早期，源码的版本控制非常容易出错，通常是以简单的目录拷贝进行数据的备份，多个程序员通过文件服务器的共享目录进行协同，公共库和头文件在开发过程中相互覆盖，非常混乱。

尽管如此，还是有非常好用的源码比较和补丁工具：diff和patch。

1.2 不同的版本控制系统

为了解决文件版本备份问题，人们很久以前就开发了许多种本地版本控制系统（图1-1），大多都是采用某种简单的数据库来记录文件的历次更新差异，后来又出现了用于多人协作开发的集中式版本控制系统（图1-2）。但这些系统的所有内容都严重依赖于中央数据库。


图1-1 本地版本控制系统


图1-1 集中式版本控制系统

后来，分布式数据库的版本控制系统（Distributed VCS，图1-3）面世，在这类系统中，诸如git、Mercurial、Bazaar还有Darcs等，客户端并不只提取最新版本的文件快照，还将代码仓库完整地镜像下来。
这么一来，任何一处协同工作的服务器发生故障，事后都可以用任何一个镜像出来的本地仓库恢复。因为每一次的提取操作，实际上都是一次对代码仓库的完整备份。其中，git是由Linus Torvalds于2005年为Linux内核开源社区所创建，已经成为最受欢迎的一个DVCS。

2. git项目的三个区域

一个git项目有三个主要的区域：工作目录、暂存区和git目录。

2.1 工作目录

工作目录，也称为工作树，是：

  • 使用git init进行git仓库初始化时的当前目录，该目录包含一个作为git仓库的.git目录
  • 以后用户添加或删除git所跟踪文件的目录
  • 用户从仓库中取出（checkout）项目某个版本的所有内容，用以开始后续开发工作的目录

在控制台下，当当前目录为工作目录时，会显示相应的分支提示符。

使用git时，当从仓库中取出项目某个版本的所有文件时，这些文件实际上都是从.git目录中的压缩对象数据库中提取出来的，接下来就可以在工作目录中使用这些文件进行开发工作。

2.1.1 工作目录的创建

工作目录是某项目的某个版本的单次签出。相应的文件从 git 目录中的压缩仓库中抽取出来并放在磁盘上,供你使用或者修改。

创建工作目录有 2 种方法:

  • 用git init命令初始化仓库,即生成.git目录;.git目录的父目录就是工作目录
  • 使用git clone命令克隆一个已有的版本库,同时也创建了相应的工作目录

2.2 git目录

git目录是存储项目的元数据和对象数据库的地方。所有的文件快照都会提交到该目录中。该目录是git最重要的部分；当从另一台计算机克隆一个仓库时，拷贝的就是这个目录。

2.3 暂存区域

顾名思义——跟踪工作目录中所有文件的状态，暂存着工作目录中某些文件的变化（使用git add来暂存变化），这些变化组成了下一次提交。

2.4 小结

一个git项目有三个区域：

  • 工作目录：该区域的文件是从仓库中签出的文件；这些文件可能处于未修改状态，也可能处于已修改但还没有快照更新状态。
  • 暂存区：文件已做快照，处于准备提交状态，存储在暂存区中。即工作区修改的文件内容会被写入到对象库中的一个新的对象中，而该对象的ID被记录在暂存区目录树的文件索引中。
  • git仓库：提交文件快照，存储在git仓库中。当执行提交操作时，暂存区的目录树会写到对象库中，master分支会指向这个新进入对象库的目录树。

3. 文件状态

工作目录下的所有文件可能处于两种状态之一：跟踪或未跟踪。

  • 跟踪文件是在上一次快照中的文件，他们可能是未修改的、修改的或暂存的。
  • 未跟踪文件是所有其他的文件，即工作目录中不在上一次快照，或者不在暂存区中的文件。当第一次克隆一个仓库时，所有的文件都是跟踪并未修改的，因为仅仅是才签出它们，还没有修改任何东西。

3.1 被跟踪文件状态

任何一个被跟踪的文件可能处于三种状态：

  • unmodified：表示文件的快照数据已被提交到仓库中，或文件刚从仓库中签出到工作目录。
  • modified：表示该文件已被修改，但还没有做快照放在暂存区中。
  • staged：表示将已修改的文件暂存起来，作为下一次提交的一部分。

4. git基本工作流程

① 在工作目录中修改某些文件；
② 暂存这些文件，即将它们的快照加入到暂存区域中；
③ 进行一次提交，获取在暂存区域中的文件，然后将该次快照永久地提交到git目录中。

如何判断文件的状态？我们可以从文件所处的位置来判断状态：

  • 如果是.git目录中保存着的特定版本文件，属于已提交状态。
  • 刚从.git仓库中签出来，但未做修改的文件，属于未修改状态。
  • 从.git仓库中签出来，做了修改但没有放到暂存区域，就是已修改状态。
  • 做了修改，快照并放入暂存区域，属于已暂存状态。

5. git对象

git有4种最基本对象：blob、tree、commit和tag。

  • blob：即文件快照，只包含内容和大小，无名字、权限等属性。
  • tree：记录着工作树内容及其中各个文件对应blob对象索引。
  • commit：包含指向tree对象的索引和其它提交信息元数据。
  • tag：标签，它可以指向blob、tree、commit并包含签名，最常见的是指向commit的GPG签名的标签。
